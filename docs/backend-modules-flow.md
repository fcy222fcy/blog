## 模块一：认证模块 (Auth)

### 模块职责
- 用户登录认证
- 用户注册
- Token 生成与验证
- 密码修改
- 用户信息获取

### 模块设计

认证模块是整个博客系统的核心入口，负责用户身份验证和权限控制。设计上采用 **JWT (JSON Web Token)** 作为认证方案，实现了无状态的用户认证机制。

**登录流程设计**：用户提交用户名和密码后，系统首先对输入进行安全验证（防止 SQL 注入等攻击），然后查询数据库验证用户身份。密码采用 bcrypt 算法加密存储，登录时通过 `bcrypt.CompareHashAndPassword` 进行安全比对。验证通过后，使用 JWT 库生成包含用户 ID 和用户名的 Token，Token 设置合理的过期时间，确保安全性。

**注册流程设计**：注册时首先检查用户名是否已存在，然后对密码进行 bcrypt 加密处理后存储到数据库。整个过程遵循"先验证后操作"的原则，确保数据一致性。

**中间件设计**：JWT 认证中间件作为所有受保护路由的守门人，负责从请求头中提取 Token、验证签名和过期时间、提取用户信息并存入 Gin Context，供后续业务逻辑使用。

**安全考虑**：输入验证防止恶意字符注入，密码加密存储防止数据泄露，Token 过期机制控制会话时效。

## 模块二：文章模块 (Article)

### 模块职责
- 文章 CRUD 操作
- 文章列表查询（前台/后台）
- 文章详情获取
- 浏览量统计（防刷）
- 文章归档
- 文章搜索

### 模块设计

文章模块是博客系统的核心内容模块，承载了博客的主要功能。设计上采用了**前后台分离**的查询策略，前台只展示已发布文章，后台可以管理所有状态的文章。

**文章模型设计**：文章实体包含标题、内容、摘要、封面、分类、标签等字段。标题使用 slug 字段作为 URL 友好的标识符，支持 SEO 优化。文章支持草稿和已发布两种状态，便于内容管理。

**浏览量防刷设计**：为了防止恶意刷量，系统实现了基于 IP 的访问控制。通过 `visit_logs` 表记录每个 IP 对文章的访问记录，24 小时内同一 IP 只计数一次。这个设计在保证统计准确性的同时，也保护了系统资源。

**阅读时长计算**：系统自动计算文章的阅读时长，采用混合算法：中文按 400 字/分钟计算，英文按 200 词/分钟计算。这个功能提升了用户体验，让读者对阅读时间有预期。

**标签关联设计**：文章与标签采用多对多关联，通过 `article_tags` 中间表实现。这种设计支持灵活的标签管理，一篇文章可以有多个标签，一个标签也可以关联多篇文章。

## 模块三：分类模块 (Category)

### 模块职责
- 分类 CRUD 操作
- 分类列表查询（带文章数量统计）
- 删除前检查关联文章

### 模块设计

分类模块用于组织和管理博客文章的分类体系。设计上采用了**树形结构**的思路，支持分类的排序和图标显示，为前端展示提供丰富的元数据。

**唯一性约束**：分类的 name 和 slug 字段都设置了唯一索引，确保分类名称和 URL 标识的唯一性。创建和更新时都会进行重复检查，避免数据冲突。

**删除保护机制**：删除分类前会检查该分类下是否有关联的文章。如果存在关联文章，则拒绝删除操作，返回错误提示。这种设计保护了数据完整性，避免了文章失去分类归属的情况。

**文章数量统计**：查询分类列表时，会同时统计每个分类下的文章数量。这个设计让前端能够展示每个分类的文章数量，帮助用户了解内容分布。

**排序支持**：分类支持 sort_order 字段，允许管理员自定义分类的显示顺序。这种设计提供了灵活的内容组织能力。

### 模块设计

标签模块与分类模块类似，但标签更侧重于**内容的多维度标记**。与分类的一对多关系不同，标签与文章是多对多关系，这种设计提供了更灵活的内容组织方式。

**多对多关联设计**：通过 `article_tags` 中间表实现文章与标签的多对多关联。这种设计允许一篇文章添加多个标签，一个标签也可以关联多篇文章，非常适合内容的交叉分类和推荐。

**删除保护**：与分类类似，删除标签前会检查关联文章数量，确保不会影响已有的文章内容。

**轻量级设计**：标签实体只包含 name 和 slug 字段，保持轻量化。这种设计让标签管理更加简单，也便于扩展。

## 模块五：评论模块 (Comment)

### 模块职责
- 评论 CRUD 操作
- 评论列表查询（支持分页）
- 评论审核（通过/拒绝）
- 评论点赞（防重复）
- 邮件通知（异步）

### 模块设计

评论模块是博客系统中交互性最强的模块，支持**嵌套回复**、**邮件通知**、**点赞防重复**等复杂功能。设计上充分考虑了用户体验和系统安全。

**嵌套评论设计**：评论支持 parent_id 字段，实现评论的嵌套回复。这种设计让讨论更加有条理，用户可以清晰地看到回复关系。查询时支持多级嵌套，前端可以渲染成树形结构。

**双模式评论**：系统同时支持注册用户评论和访客评论。注册用户通过 user_id 关联，访客通过 visitor_id 关联。这种设计既保护了用户体验，又降低了评论门槛。

**邮件通知机制**：评论创建后，系统会异步发送邮件通知。如果是回复评论，通知被回复的用户；如果是新评论，通知博主。使用 goroutine 异步处理，不阻塞主流程。

**点赞防重复**：通过 `comment_like_logs` 表记录点赞日志，每个 IP 对同一评论只能点赞一次。这种设计保证了点赞数据的真实性。

**审核机制**：评论支持 pending/approved/rejected 三种状态，管理员可以审核评论内容，确保社区环境。

## 模块八：媒体模块 (Media)

- 文件上传（图片、文档）
- 文件列表查询
- 文件删除

### 模块设计

媒体模块负责博客系统的文件上传和管理，设计上**以安全性为核心**，实现了多层文件验证机制。

**安全验证链**：文件上传时会经过四层验证：文件大小检查（≤5MB）、扩展名白名单（.jpg/.jpeg/.png/.gif/.webp/.pdf）、Content-Type 验证（防止扩展名伪造）、文件名安全检查（防止路径遍历）。这种设计确保了上传文件的安全性。

**文件系统存储**：当前采用本地文件系统存储，文件保存在 `./uploads` 目录下。这种设计简单高效，适合博客系统的规模。后续可以扩展为 OSS 对象存储。

**文件列表管理**：支持分页查询和关键词过滤，方便管理员管理上传的文件。文件信息包括名称、URL、大小、类型、修改时间等。

## 模块九：每日一问模块 (DailyQuestion)

### 模块职责
- 每日一问 CRUD 操作
- 问题列表查询
- 问题点赞
- 浏览量统计

### 模块设计

每日一问模块是博客的特色功能，设计上以**日期为核心**，每天发布一个问题，鼓励用户参与讨论。

**日期唯一性**：每个问题都有唯一的日期标识，同一天只能有一个问题。这种设计保证了"每日一问"的核心概念，避免了重复发布。

**时间导航**：支持按日期查询问题，同时提供前一天和后一天的导航功能。这种设计让用户可以轻松浏览历史问题，增加了内容的可发现性。

**互动统计**：问题支持点赞、浏览量、评论数等统计。这些数据帮助管理员了解用户兴趣，优化内容策略。

**状态管理**：问题支持启用/禁用状态，管理员可以控制问题的显示。这种设计提供了灵活的内容管理能力。

关于页面模块用于管理博客的"关于"页面内容，设计上采用了**单页内容**的思路，简单高效。

## 模块十二：RSS/Sitemap 模块

### 模块职责
- RSS 订阅生成
- Sitemap 生成

### 模块设计

RSS/Sitemap 模块为博客提供**标准化的内容分发**能力，设计上遵循 RSS 2.0 和 Sitemap 协议规范。

**RSS 订阅**：生成标准的 RSS 2.0 XML 格式，包含博客的基本信息和最新的文章列表。RSS 阅读器可以自动订阅，帮助用户及时获取更新。

**动态生成**：RSS 和 Sitemap 内容动态生成，每次请求都查询最新的文章数据。这种设计保证了内容的实时性。
