# 博客搜索功能设计文档

## 1. 功能概述

本站搜索功能覆盖**前台搜索结果页**（`/search`）和**全局弹窗快速搜索**（侧边栏/移动端的搜索按钮），支持对已发布文章的**标题、正文、摘要**三字段进行模糊匹配搜索，并在结果中对命中的关键词进行高亮显示。当关键词仅出现在正文时，会从正文中抽取包含该关键词的上下文片段（带 `[...]` 省略号）作为搜索摘要，避免仅显示毫无相关性的文章自带 Summary。

### 3.2 Repository 层：数据库搜索

位置：[article_repository.go:Search](file:///e:/goCode/goFile/gin博客/internal/repository/article_repository.go#L229-L253)

```sql
SELECT * FROM articles
WHERE status = 'published'
  AND (title   LIKE '%<keyword>%'
    OR content LIKE '%<keyword>%'
    OR summary LIKE '%<keyword>%')
ORDER BY is_top DESC, created_at DESC
OFFSET <offset> LIMIT <limit>;
```

要点：
- 仅搜 `status='published'`（草稿、定时发布不对外暴露）；
- 三字段 OR 模糊匹配（注意：MySQL LIKE 对中文一般走全表扫描，数据量大后建议升级为 FULLTEXT 索引或 Elasticsearch/Meilisearch）；
- 固定排序：置顶优先，再按发布时间倒序（后续可扩展按相关度排序）；
- 自动 `Preload("Category")` 与 `Preload("Tags")`，与首页/详情页数据形态一致。

#### 3.3.2 核心算法 1：`stripMarkdown` — 清洗 Markdown 保留纯文本

位置：[article_service.go:stripMarkdown](file:///e:/goCode/goFile/gin博客/internal/service/article_service.go#L465-L514)

将 Markdown 原文转为纯文本，供后续在正文中找关键词匹配位置。**清洗策略由外到内逐条剥离，避免 Go 的 RE2 不支持反向引用引起的嵌套正则错误**：

| 步骤 | 处理项 | 替换为 | 说明 |
|---|---|---|---|
| 1 | 围栏代码块 ` ``` ... ``` ` | ` $1 ` （$1=代码内容，语言名丢弃） | ⚠️ 历史 BUG 曾直接删成空串，导致代码块内关键词丢失，搜索摘要无法生成（已修复） |
| 2 | 行内代码 `` `xxx` `` | ` $1 `（保留内容，去反引号） | ⚠️ 同上曾直接删除内容 |
| 3 | HTML 注释 `<!-- ... -->` | 删除 | 无业务语义 |
| 4 | HTML 标签 `<xxx>` | 删除 | 仅保留可见文字 |
| 5 | 图片 `![alt](url)` | `$1`(alt 文本) | 图片里的文字也可能有关键词 |
| 6 | 链接 `[text](url)` | `$1`(链接文字) | 去 URL 保留文字 |
| 7 | 行前缀 `#`/`>`/`- `/`1. ` | 删除 | 去标题/引用/列表标记 |
| 8 | `***` → `**` → `*` 多层加粗/斜体 | 最外层先剥，循环到无 | 避免 RE2 无反向引用 |
| 9 | `~~~` / `~~` 多层删除线 | 同上 | |
| 10 | `___` / `__` / `_` 多层下划线 | 同上 | 含边界断言避免英文单词中间下划线 |
| 11 | 残留的 `*` / `~` | 删除（strings.Map） | 兜底 |
| 12 | 连续空白 `\s+` | 单个空格 | 归一化空白 |

所有正则由 `compileRegexps()` 通过 `sync.Once` 一次性编译，`mustOrEmpty` 正则编译失败时返回一个永不命中的 `a^` 空壳正则，**避免启动时 panic**，保证服务可用。

#### 3.3.3 核心算法 2：`keywordPattern` — 多关键词 OR 匹配正则

位置：[article_service.go:keywordPattern](file:///e:/goCode/goFile/gin博客/internal/service/article_service.go#L516-L533)

- 用 `strings.Fields(keyword)` 按空白拆成多个关键词；
- 每个关键词经过 `regexp.QuoteMeta(k)` 转义，避免用户输入 `.*`/`[abc]` 等特殊字符引发 RE 语法问题或意外匹配；
- 最后用 `|` 拼成 `(?i)(kw1|kw2|...)`，**不区分大小写**（中文无影响，英文搜索更好用）。

#### 3.3.4 核心算法 3：`buildSearchSnippet` — 抽取关键词上下文片段

位置：[article_service.go:buildSearchSnippet](file:///e:/goCode/goFile/gin博客/internal/service/article_service.go#L535-L605)

**函数签名**：`buildSearchSnippet(content, summary, keyword string, radius int) string`
- `radius` 为关键词前后保留的 rune 数（中文近似汉字数），搜索服务中传 `snippetRadius=45`，即上下约各 45 个汉字，共 ~90 字片段。

**执行流程**：

```
keyword 为空 → 返回 ""
    │
    ▼
Summary 中是否已命中任一关键词？
    ├─ 是 → 返回 ""（前端直接走 Summary 即可，避免重复截断）
    └─ 否
        │
        ▼
对 content 运行 stripMarkdown → clean
    │
    ├─ clean 为空 → 返回 ""
    └─ 否则运行 keywordPattern 正则
              │
              ├─ 正则构建失败 → 返回 ""
              └─ re.FindStringIndex(clean) 找第一个命中的 byte 区间 loc
                        │
                        ├─ loc == nil（命中位置不存在，理论上极少发生，
                        │            因为 repo 已 content LIKE 命中，可能仅因
                        │            Markdown 清洗后丢失所致）→ 返回 ""
                        └─ 将 byte 位置 loc 映射为 rune 位置 [hitStart, hitEnd)
                                   │（Go string 是 UTF-8 byte 序列，中文 1 字
                                   │  = 3 byte，正则返回的是 byte 下标，必须
                                   │  逐 rune 扫描才能正确做"汉字级切片"）
                                   ▼
                    左边界 left  = max(0, hitStart - radius)
                    右边界 right = min(len(runes), hitEnd + radius)
                                   │
                                   ▼
                    组装：[ left>0 ? "[...]" : "" ] + segment
                         + [ right<len(runes) ? "[...]" : "" ]
                                   │
                                   ▼
                              返回最终片段字符串
```

关于 byte→rune 映射（L561-L577）：
```go
runes := []rune(clean)
hitStart, hitEnd := -1, -1
cur := 0
for i := 0; i < len(runes); i++ {
    next := cur + len(string(runes[i:i+1])) // 第 i 个 rune 的 byte 长度
    if cur <= loc[0] && loc[0] < next && hitStart == -1 { hitStart = i }
    if cur <= loc[1] && loc[1] <= next                 { hitEnd   = i + 1 }
    if hitStart >= 0 && hitEnd >= 0 { break }
    cur = next
}
```
这是 `search_snippet` 能"汉字对齐、不切断半个中文字符"的关键保证。

### 4.3 关键词高亮渲染（防 XSS 安全实现）

位置：[Search.vue:highlightTokens](file:///e:/goCode/goFile/gin博客/blog-web/src/views/Search.vue#L120-L146)

**核心设计：不用 v-html，把文本切成 `<普通段>` / `<命中段>` token 数组交给 Vue 模板渲染，从根本上避免 XSS。**

```js
const highlightTokens = (text, kw) => {
  const src = String(text ?? '')
  const key = String(kw ?? '').trim()
  if (!src || !key) return [{ text: src, matched: false }]
  const keys = key.split(/\s+/).filter(Boolean)
  const pattern = new RegExp(keys.map(escapeRegExp).join('|'), 'ig')
  const out = []
  let lastEnd = 0
  let m
  while ((m = pattern.exec(src)) !== null) {
    if (m.index > lastEnd) out.push({ text: src.slice(lastEnd, m.index), matched: false })
    out.push({ text: m[0], matched: true })
    lastEnd = m.index + m[0].length
    if (m.index === pattern.lastIndex) pattern.lastIndex++ // 零宽防死循环
  }
  if (lastEnd < src.length) out.push({ text: src.slice(lastEnd), matched: false })
  if (out.length === 0) out.push({ text: src, matched: false })
  return out
}
```

辅助函数 `escapeRegExp(s)` 转义正则特殊字符，与后端 `regexp.QuoteMeta` 对应。

模板侧使用（标题 / 摘要各一套）：
```html
<!-- 标题高亮 -->
<h3 class="article-list-title">
  <template v-for="(seg, idx) in highlightTokens(article.title, keyword)" :key="'t-'+article.id+'-'+idx">
    <mark v-if="seg.matched" class="search-hl">{{ seg.text }}</mark>
    <span v-else>{{ seg.text }}</span>
  </template>
</h3>

<!-- 摘要高亮 -->
<p class="article-list-summary">
  <template v-for="(seg, idx) in highlightTokens(displaySummary(article), keyword)" :key="'s-'+article.id+'-'+idx">
    <mark v-if="seg.matched" class="search-hl">{{ seg.text }}</mark>
    <span v-else>{{ seg.text }}</span>
  </template>
</p>
```

样式类 `.search-hl` 在全局 CSS 中定义为"背景半透明强调色 + 原文字色不变"的柔和高亮风格。

## 6. 历史问题 / BUG 修复记录

### 6.1 搜索结果无法高亮（代码块/行内代码关键词丢失）

- **问题现象**：搜索"文章"，文章 A 明明出现在结果里，但标题和摘要都没"文章"，完全不高亮，用户觉得"搜错了"
- **根因**：`stripMarkdown` 里 `reFenced` 和 `reInlineCode` 的正则写成匹配整段 ` ```...``` ` 和 `` `xxx` ``，然后 `ReplaceAllString(s, "")` **连内容带标记一起删了**。由于数据库 LIKE 用的是原始 Markdown（保留代码内容），所以 repo 能命中；但 buildSearchSnippet 找位置用的是清洗后纯文本，代码块里的关键词被删干净，re.FindStringIndex 返回 nil → `search_snippet` 空 → 前端走 Summary（没关键词）→ 不高亮。
- **修复**：
  - `reFenced` 改为带捕获组：`(?s)\`\`\`[^\n]*\n?(.*?)\`\`\``，Replace 为 ` $1 `（只去 ```和语言行，保留代码文字）；
  - `reInlineCode` 改为带捕获组：`` `([^`]*)` ``，Replace 为 ` $1 `（只去反引号，保留行内文字）。
- **位置**：[article_service.go:compileRegexps](file:///e:/goCode/goFile/gin博客/internal/service/article_service.go#L434-L452) + [article_service.go:stripMarkdown](file:///e:/goCode/goFile/gin博客/internal/service/article_service.go#L471-L473)

## 7. 可扩展方向

| 方向 | 说明 |
|---|---|
| 相关度排序 | 目前按置顶+时间；可引入 `title命中>summary命中>content命中`、命中次数加权等打分公式，替代 ORDER BY |
| 全文索引 | MySQL LIKE 无法利用索引，数据量大时升级 FULLTEXT（`MATCH AGAINST`）或接入 MeiliSearch / Elasticsearch |
| 高亮片段多命中 | `buildSearchSnippet` 目前只取第一个命中，可扩展为最多显示 N 个命中片段，用 `...` 分隔，类似 Google / liuhouliang.com |
| 搜索联想/纠错 | 在 SearchModal 加热门词、历史搜索（localStorage）、拼写提示 |
| 搜索统计 | 接入埋点记录无结果词（热门无结果词=SEO内容选题） |
| 多语言分词 | 英文词干化、中文 jieba/结巴分词，解决"文章"搜不到"文章们"/"这篇文章"之类匹配需求 |
