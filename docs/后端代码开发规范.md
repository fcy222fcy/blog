# 后端代码开发规范

# 一、整体架构规范

项目采用 `cmd / internal / pkg` 分层：

```text
cmd/server/main.go          程序入口，只负责启动 App
internal/app/app.go         应用启动器，统一初始化配置、日志、数据库、依赖、路由、HTTP Server
internal/api                Controller/API 层
internal/service            Service 业务层
internal/repository         Repository 数据访问层
internal/model              Entity 和 DTO
internal/middleware         Gin 中间件
pkg                         可复用基础能力，如 config、logger、database、jwt、response、errors
configs                     配置文件
scripts                     数据库脚本
docs                        架构和接口文档

```

核心调用链是：

```text
HTTP Request
 -> Middleware
 -> Controller
 -> Service
 -> Repository
 -> Database
 -> Repository
 -> Service
 -> Controller
 -> Unified Response

```

禁止跨层调用。比如 Controller 不直接访问 Repository，Repository 不处理业务规则。

# 二、MVC / 三层职责规范

**Controller 层：**`**internal/api/v1/{module}**`

职责：

*   只处理 HTTP 相关逻辑。
    
*   使用 `ShouldBindJSON` / `ShouldBindQuery` 绑定和校验请求参数。
    
*   从中间件上下文读取用户信息，比如 `middleware.GetUserID(c)`。
    
*   调用 Service 完成业务。
    
*   使用 `pkg/response` 返回统一响应。
    
*   不写业务规则，不直接访问数据库。
    

典型结构：

```go
type Controller struct {
    userService service.UserService
}

func NewController(userService service.UserService) *Controller {
    return &Controller{userService: userService}
}

```

路由单独放在 `routes.go`：

```go
func (ctrl *Controller) RegisterRoutes(r *gin.RouterGroup) {
    userGroup := r.Group("/user")
    userGroup.Use(middleware.Auth())
    {
        userGroup.GET("/profile", ctrl.GetProfile)
    }
}

```

**Service 层：**`**internal/service**`

职责：

*   承载核心业务逻辑。
    
*   做业务校验，比如用户是否存在、密码是否正确、状态是否禁用。
    
*   调用 Repository 完成数据操作。
    
*   做 DTO / Entity 转换。
    
*   处理分页参数标准化。
    
*   需要事务时，事务应该放在 Service 层。
    

接口和实现分离：

```text
user_interface.go   定义 UserService 接口
user_service.go     实现 userService

```

规范：

```go
type UserService interface {
    Register(req *request.RegisterRequest) error
    GetUserByID(id uint) (*entity.User, error)
}

type userService struct {
    userRepo repository.UserRepository
}

func NewUserService(userRepo repository.UserRepository) UserService {
    return &userService{userRepo: userRepo}
}

```

实现结构体小写，接口大写导出。外部只能通过构造函数拿到接口。

**Repository 层：**`**internal/repository**`

职责：

*   封装数据库访问。
    
*   使用 GORM 查询、创建、更新、删除。
    
*   不写复杂业务规则。
    
*   查询不到数据时返回 `nil, nil`，由 Service 转换成业务错误。
    
*   分页查询返回 `list, total, error`。
    

接口和实现分离：

```text
user_interface.go       定义 UserRepository
user_repository.go      实现 userRepository

```

典型规范：

```go
type UserRepository interface {
    FindByID(id uint) (*entity.User, error)
    Create(user *entity.User) error

    List(offset, limit int) ([ ]*entity.User, int64, error)

}

type userRepository struct {
    db *gorm.DB
}

func NewUserRepository(db *gorm.DB) UserRepository {
    return &userRepository{db: db}
}

```

# 三、依赖注入规范

项目采用 **构造函数依赖注入**，没有使用全局 Service 或 Repository 单例。

依赖方向：

```text
Controller 依赖 Service 接口
Service    依赖 Repository 接口
Repository 依赖 *gorm.DB

```

统一在 `internal/app/app.go` 的 `initDependencies()` 里装配：

```go
userRepo := repository.NewUserRepository(a.mysqlDB)

userSvc := service.NewUserService(userRepo)
authSvc := service.NewAuthService(userRepo, userSvc)

a.router = api.NewRouter(userSvc, authSvc)

```

优点：

*   方便单元测试 Mock。
    
*   模块边界清晰。
    
*   新增模块时只需要新增 Repository、Service、Controller，并在 App 中注册。
    

# 四、接口定义规范

每个模块通常拆成两类接口：

Repository 接口：

```go
type XxxRepository interface {
    FindByID(id uint) (*entity.Xxx, error)

    List(offset, limit int) ([ ]*entity.Xxx, int64, error)

    Create(x *entity.Xxx) error
    Update(x *entity.Xxx) error
    Delete(id uint) error
}

```

Service 接口：

```go
type XxxService interface {
    Create(req *request.CreateXxxRequest) error
    GetByID(id uint) (*entity.Xxx, error)
    Update(id uint, req *request.UpdateXxxRequest) error
    Delete(id uint) error
    List(req *request.XxxListRequest) (*response.PageResponse, error)
}

```

接口命名按业务能力命名，如：

*   `UserRepository`
    
*   `UserService`
    
*   `AuthService`
    

实现结构体使用小写：

*   `userRepository`
    
*   `userService`
    
*   `authService`
    

构造函数统一：

```go
NewXxxRepository(...)
NewXxxService(...)
NewController(...)

```

# 五、模型规范

模型分为 Entity 和 DTO。

Entity 放在：

```text
internal/model/entity

```

用于数据库映射，带 GORM 标签：

```go
type User struct {
    BaseEntity
    Username string `gorm:"type:varchar(50);uniqueIndex;not null" json:"username"`
    Password string `gorm:"type:varchar(255);not null" json:"-"`
}

```

每个实体建议实现表名：

```go
func (User) TableName() string {
    return "users"
}

```

公共字段放在 `BaseEntity`：

```go
ID
CreatedAt
UpdatedAt
DeletedAt

```

DTO 放在：

```text
internal/model/dto/request
internal/model/dto/response

```

Request DTO 负责接收参数和校验：

```go
type RegisterRequest struct {
    Username string `json:"username" binding:"required,min=3,max=50"`
    Password string `json:"password" binding:"required,min=6,max=50"`
}

```

Response DTO 负责返回字段，避免直接暴露 Entity 的敏感字段，比如密码 `json:"-"`。

# 六、路由规范

总路由在：

```text
internal/api/router.go

```

模块路由在：

```text
internal/api/v1/{module}/routes.go

```

路由分组规范：

```text
/api/v1/auth/register
/api/v1/auth/login
/api/v1/user/profile

```

认证类接口挂在 `/auth`，业务模块接口挂在自己的模块路径下。

需要登录的模块使用：

```go
userGroup.Use(middleware.Auth())

```

# 七、统一响应规范

响应统一使用 `pkg/response`。

成功响应：

```json
{
  "code": 0,
  "message": "成功",
  "data": {}
}

```

错误响应也是 HTTP 200，业务错误通过 `code` 区分：

```json
{
  "code": 1001,
  "message": "用户不存在"
}

```

分页响应格式：

```json
{

  "list": [ ],
  "total": 100,
  "page": 1,
  "size": 10,
  "total_page": 10
}

```

Controller 中不要直接 `c.JSON` 返回业务数据，统一使用：

```go
response.Success(c, data)
response.BadRequest(c, err.Error())
response.BizError(c, err)
response.Unauthorized(c, "message")

```

# 八、错误处理规范

错误定义集中在：

```text
pkg/errors/code.go
pkg/errors/errors.go

```

错误码分段：

```text
0       成功
400-499 通用客户端错误
500-599 服务端错误
1000+   用户 / 认证业务错误
2000+   参数错误
3000+   资源错误

```

Service 层返回业务错误：

```go
return bizerrors.ErrUserNotFound
return bizerrors.New(bizerrors.CodeUserAlreadyExists, "邮箱已被注册")

```

Controller 层统一转换：

```go
response.BizError(c, err)

```

# 九、中间件规范

中间件放在：

```text

```

已有中间件：



# 十一、新增业务模块标准流程

新增一个 `product` 模块时，按这个顺序：

1.  新增 Entity  `internal/model/entity/product.go`
    
2.  新增 Request / Response DTO  `internal/model/dto/request/product.go``internal/model/dto/response/product.go`
    
3.  新增 Repository 接口和实现  `internal/repository/product_interface.go``internal/repository/product_repository.go`
    
4.  新增 Service 接口和实现  `internal/service/product_interface.go``internal/service/product_service.go`
    
5.  新增 Controller 和 routes  `internal/api/v1/product/controller.go``internal/api/v1/product/routes.go`
    
6.  在 `internal/api/router.go` 注册 Controller 路由。
    
7.  在 `internal/app/app.go` 的 `initDependencies()` 注入 Repository / Service。
    
8.  在 `initDatabase()` 的 `AutoMigrate` 中加入新 Entity。

# 十三、这个模板的开发原则

可以总结成几条硬规范：

1.  Controller 只处理 HTTP，不写业务。
    
2.  Service 写业务规则，不直接暴露给 Gin。
    
3.  Repository 只处理数据访问，不写业务判断。
    
4.  所有依赖通过构造函数注入。
    
5.  Service 和 Repository 必须定义接口。
    
6.  请求和响应必须使用 DTO，不直接把数据库 Entity 当 API 契约。
    
7.  响应必须走统一 `response` 包。
    
8.  业务错误必须走统一 `pkg/errors`。
    
9.  新模块必须按 Entity、DTO、Repository、Service、Controller、Routes、App 注入的顺序添加。
    
10.  全局初始化逻辑集中在 `internal/app/app.go`，`main.go` 保持极简。
